# Level 5: Load balancers: spreading the work across many machines

**Track:** System Design · **Level:** 5/12 · **Difficulty:** `intermediate`

📚 **Today's lesson** — published 2026-06-28

## TL;DR

A load balancer sits in front of your servers and routes each request to a healthy one. It's how you go from 'one server handling everything' to 'hundreds of servers sharing the load'.

## Real-world analogies

- A load balancer is a supermarket cashier dispatcher. Customers (requests) come in. The dispatcher sends them to whichever checkout (server) has the shortest line.
- Without a load balancer, all your customers would queue at one cashier. Even if you have 10 cashiers waiting.

## Key concepts

### `L4 vs L7 load balancing`

L4 looks at IP+port (fast, simple). L7 looks at the HTTP request (smart, can route by URL or cookie).

### `Algorithms`

Round robin (next server), least connections (server with fewest), IP hash (same user → same server), weighted (some servers are bigger).

### `Health checks`

The load balancer pings each server. If one stops responding, traffic stops going to it. Critical for reliability.

### `Sticky sessions`

Some apps need the same user to land on the same server (because they store session in memory). Modern apps avoid this — store sessions in Redis.

## Try it yourself

You have 3 app servers and a load balancer. One server starts failing 50% of requests. What happens? How do you detect it? How do you recover?

## Common pitfalls

- ⚠️ Single load balancer = single point of failure. Use two LBs in active-passive or DNS-level round-robin.
- ⚠️ Forgetting to drain connections on deploy. Server gets killed mid-request, user sees error.
- ⚠️ Health checks that are too strict. App is fine, DB is slow → health check fails → LB removes it → cascade failure.
