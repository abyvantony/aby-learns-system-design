# Level 4: Caching: the fastest way to make things fast

**Track:** System Design · **Level:** 4/12 · **Difficulty:** `intermediate`

📚 **Today's lesson** — published 2026-06-28

## TL;DR

Caching stores frequently-accessed data in memory (fast) instead of recomputing or re-reading from disk (slow). Done right, it's the single biggest performance win. Done wrong, it causes stale data bugs.

## Real-world analogies

- Cache is your desk. Books on it are fast to grab. Books in the library are slower. If a book gets updated in the library, your desk version is stale until you replace it.
- Browser cache is like a fridge. You keep common things (milk, eggs) close. When you run out, you go to the store (origin server).

## Key concepts

### `Cache levels`

Browser → CDN → Load Balancer → Application server → Database. Each level caches different things.

### `Cache invalidation`

The hardest problem in CS (alongside naming things). When do you throw away the cached value? On TTL? On update?

### `Cache-aside vs write-through`

Cache-aside: app checks cache, falls back to DB, populates cache. Write-through: every write goes to cache + DB.

### `Cache stampede`

When a popular cache key expires, 1000 requests hit the DB at once. Solve with locking, request coalescing, or probabilistic early expiration.

```
# Cache-aside pattern (most common)

def get_user(user_id):
    # 1. Check cache first
    cached = redis.get(f"user:{user_id}")
    if cached:
        return json.loads(cached)  # Hit

    # 2. Cache miss — go to DB
    user = db.query("SELECT * FROM users WHERE id = ?", user_id)

    # 3. Populate cache for next time
    if user:
        redis.setex(f"user:{user_id}", 3600, json.dumps(user))  # 1 hour TTL

    return user

# Write-through: update cache when you update DB
def update_user(user_id, name):
    db.execute("UPDATE users SET name = ? WHERE id = ?", name, user_id)
    user = db.query("SELECT * FROM users WHERE id = ?", user_id)
    redis.setex(f"user:{user_id}", 3600, json.dumps(user))

# === Cache eviction policies ===
# LRU (Least Recently Used): throw out the entry not accessed for the longest
# LFU (Least Frequently Used): throw out the entry accessed the fewest times
# TTL (Time To Live): throw out entries after N seconds
# FIFO (First In, First Out): throw out the oldest entry

# === What to cache ===
# - Read-heavy data (user profiles, product info)
# - Expensive computations (search results, recommendations)
# - API responses from external services (rate limits!)
#
# === What NOT to cache ===
# - Frequently changing data (stock prices, live scores)
# - User-specific data with high cardinality (every user's session)
# - Anything you can't invalidate cleanly

# === Cache metrics to watch ===
# Hit rate = hits / (hits + misses)
#   > 90% = great
#   70-90% = OK
#   < 70% = cache is too small or your keys are wrong
```

## Common pitfalls

- ⚠️ Caching without thinking about invalidation. 'But it works on my machine' — until the data changes and users see the old version.
- ⚠️ Caching data that's personalized without including the user ID in the key. User A sees User B's data. Disaster.
- ⚠️ Storing too much in cache. Redis is in memory = expensive. Don't put 10GB of images in Redis.
