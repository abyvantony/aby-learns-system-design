# Level 3: Databases: SQL vs NoSQL, and when to use which

**Track:** System Design · **Level:** 3/12 · **Difficulty:** `beginner-plus`

📚 **Today's lesson** — published 2026-06-28

## TL;DR

Databases are where your data lives. SQL databases (PostgreSQL, MySQL) are like Excel on steroids — structured tables with relationships. NoSQL databases (MongoDB, Redis, Cassandra) are flexible — they store data in formats closer to what your app uses.

## Real-world analogies

- SQL is a spreadsheet with superpowers. Tables are sheets, rows are entries, columns are fields. You can JOIN sheets to combine data.
- NoSQL is a filing cabinet where each folder can have different sections. Faster to drop new types of stuff in, harder to query across folders.
- Redis (a key-value store) is a coat check. You hand over your coat (value) and get a ticket (key). Later, give the ticket back, get your coat. No searching through racks.

## Key concepts

### `ACID`

Atomicity, Consistency, Isolation, Durability. SQL databases guarantee these — your transaction either fully completes or doesn't happen at all.

### `CAP theorem`

You can have only 2 of: Consistency, Availability, Partition tolerance. In practice, networks always fail, so it's CA or AP.

### `Vertical vs horizontal scaling`

Vertical: bigger server. Horizontal: more servers. SQL scales up well, NoSQL scales out well.

### `Indexing`

Like a book index — speeds up lookups by a column. Slows down writes. Don't index everything.

### `Replication`

Copy data to multiple servers. Read replicas serve reads; primary handles writes. Improves read speed and availability.

## Common pitfalls

- ⚠️ Choosing NoSQL just because it's trendy. SQL handles 90% of use cases just fine.
- ⚠️ Not adding indexes on foreign keys. JOINs on unindexed columns = full table scan = slow.
- ⚠️ Using MongoDB like a SQL database with rich queries. If you need complex JOINs, use PostgreSQL.
