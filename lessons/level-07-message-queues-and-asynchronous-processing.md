# Level 7: Message queues and asynchronous processing

**Track:** System Design · **Level:** 7/12 · **Difficulty:** `intermediate-plus`

📚 **Today's lesson** — published 2026-06-29

## TL;DR

Message queues decouple parts of your system. Instead of one service calling another directly, it drops a message in a queue. The other service picks it up when it's ready. This makes systems more resilient and scalable.

## Real-world analogies

- A message queue is a post office. You drop a letter in the mailbox (publish). The postal worker takes it to the recipient (consume) when they're home. You don't wait for the recipient to open the door.
- Restaurant orders: the waiter writes your order on a ticket and pins it to the rail. The kitchen picks tickets off the rail when they have capacity. The waiter doesn't stand at the kitchen door.

## Key concepts

### `Pub/sub vs work queue`

Pub/sub: many consumers get each message (notifications). Work queue: one consumer gets each message (tasks).

### `At-least-once delivery`

The default. A message might be delivered more than once if the consumer crashes mid-processing. Idempotency is the consumer's job.

### `Dead letter queue`

Where messages go when they fail repeatedly. Don't lose them silently — alert on messages landing here.

### `Backpressure`

What happens when producers are faster than consumers. The queue grows. Monitor queue depth. If it grows unbounded, you'll OOM.

# === Direct call (synchronous, brittle) ===
def place_order(order):
    charge_card(order)        # 200ms
    send_email(order)         # 300ms
    update_inventory(order)   # 100ms
    notify_warehouse(order)   # 500ms
    # Total: 1.1 seconds. If email service is down, order fails.

# === With message queue (async, resilient) ===
def place_order(order):
    save_to_db(order)
    queue.publish('order.placed', order)
    return "Order received!"  # Total: 50ms

# Separately, a worker:
def handle_order_placed(order):
    charge_card(order)
    queue.publish('order.charged', order)

def handle_order_charged(order):
    send_email(order)
    update_inventory(order)
    notify_warehouse(order)

# Each step is independent. If email is down, the email message waits
# in the queue. Order is still placed.

# === Common queue technologies ===
# RabbitMQ: feature-rich, complex, AMQP protocol
# Redis Streams: simple, fast, good for low-volume
# Apache Kafka: high throughput, log-based, replayable
# AWS SQS: managed, simple, pay per message
# Google Pub/Sub: managed, global, push or pull

# === Kafka example (simplified) ===
# Producer:
#   producer.send('orders', key=order.id, value=order.json())
#
# Consumer:
#   for message in consumer:
#       order = json.loads(message.value)
#       process(order)
#       consumer.commit()  # Mark as processed

# === Order of guarantees matters ===
# If 'order.charged' is published before 'order.charged.email' finishes,
# the email step might never see the message. Solutions:
#   - Saga pattern (compensating transactions)
#   - Outbox pattern (write events to DB, then publish)
#   - Eventual consistency: accept that things happen "eventually"

## Common pitfalls

- ⚠️ Treating the queue as the source of truth. Queues are for messages, not data. Your DB is the source of truth.
- ⚠️ Not making consumers idempotent. If the same message is delivered twice (it happens), you might process the payment twice.
- ⚠️ Forgetting to set TTLs on messages. A stuck message in the queue for 30 days is usually useless — and a memory leak.
