# Level 10: Monitoring, observability, and SRE basics

**Track:** System Design · **Level:** 10/12 · **Difficulty:** `advanced`

📚 **Today's lesson** — published 2026-07-01

## TL;DR

Building the system is half the job. The other half is knowing when it's broken. Monitoring tells you the system's vitals. Observability lets you ask 'why' something broke. Both are essential for production systems.

## Real-world analogies

- Monitoring is a car's dashboard. Speedometer, fuel gauge, engine light. You see what's happening now.
- Observability is the mechanic's diagnostic tools. Plug in, see the engine's full state, figure out why the light is on.
- Logging is the black box. Records of what happened. Use it after the fact to reconstruct events.

## Key concepts

### `The three pillars`

Metrics (numbers over time: CPU, latency, requests/sec), Logs (discrete events with context), Traces (request journey across services).

### `SLO/SLI/SLA`

SLI: the metric (99% of requests < 200ms). SLO: the target (we aim for 99% of requests < 200ms). SLA: the contract (we promise 99.9% uptime or your money back).

### `Alerting`

Get notified when something needs attention. Avoid alert fatigue — alert on symptoms (user-facing), not causes (CPU 80%).

### `Error budgets`

If your SLO is 99.9%, you can be down 43 minutes/month. Spend the budget on deploys. Run out → freeze deploys until you fix reliability.

```
# === Metrics: numbers you record over time ===
# Counter: things that only go up (requests, errors, bytes sent)
# Gauge: things that go up and down (current connections, queue depth)
# Histogram: distribution (latency, response sizes)

# Example: record a request
# metrics.counter('http_requests_total', {
#     'method': 'GET',
#     'status': 200,
#     'path': '/api/users'
# }).inc()
#
# metrics.histogram('http_request_duration_ms', duration_ms, {
#     'method': 'GET',
#     'path': '/api/users'
# }).observe()

# === Structured logs ===
# {
#   "timestamp": "2026-06-27T10:23:45.123Z",
#   "level": "error",
#   "service": "payment-service",
#   "trace_id": "abc123",
#   "user_id": 42,
#   "message": "Payment failed",
#   "error": "card_declined",
#   "amount_cents": 5000
# }
# Always include: timestamp, level, service, request/trace ID, structured fields

# === Distributed tracing ===
# A single user request might hit: nginx → api → auth → payment → stripe
# Each step is a 'span'. A trace is the tree of all spans.
# Tools: Jaeger, Zipkin, OpenTelemetry, Datadog APM
#
# Trace example:
# Trace abc123 (350ms total)
# ├── nginx (5ms)
# ├── api-gateway (10ms)
# ├── auth-service (50ms) ← slow!
# ├── payment-service (200ms)
# │   ├── stripe-api (180ms) ← really slow
# │   └── db-write (15ms)
# └── notification-service (85ms)

# === Golden signals (Google SRE) ===
# Latency: how long requests take
# Traffic: requests per second
# Errors: rate of failures
# Saturation: how full the system is (CPU, memory, queue depth)

# === USE method (per resource) ===
# Utilization: % of time resource is busy
# Saturation: queue length or wait time
# Errors: error count

# === RED method (per service) ===
# Rate: requests per second
# Errors: failures per second
# Duration: how long requests take

# === Alerting rules ===
# DO alert on:
#   - SLO breach (99% < 99.9% over 1 hour)
#   - Error rate spike (5xx > 1% for 5 min)
#   - Saturation (disk > 90%)
#
# DON'T alert on:
#   - CPU > 80% (symptom, often not user-facing)
#   - Single instance down (load balancer handles it)
#   - Single error (could be a transient)
```

## Try it yourself

Pick a system you built (or use daily). Define 3 SLIs, 3 SLOs, and the alerts you'd set up. Be specific: '99% of logins complete in < 1 second' is good. 'App is fast' is not.

## Common pitfalls

- ⚠️ Metrics without logs. 'Latency spiked' — why? You need logs/traces to answer.
- ⚠️ Alerting on causes. CPU at 80% doesn't matter if users are happy. Alert on what users see.
- ⚠️ No runbook for alerts. Page at 3 AM, no one knows what to do. Every alert needs a runbook link.
