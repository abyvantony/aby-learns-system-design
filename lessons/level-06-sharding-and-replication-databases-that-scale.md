# Level 6: Sharding and replication: databases that scale

**Track:** System Design · **Level:** 6/12 · **Difficulty:** `intermediate`

📚 **Today's lesson** — published 2026-06-29

## TL;DR

When one database server can't handle the load or storage, you have two options: replicate (copy data to multiple servers) or shard (split data across servers). Replication scales reads. Sharding scales writes and storage.

## Real-world analogies

- Replication is photocopying a book. Everyone gets the same book. You can read from any copy, but writes go to the master.
- Sharding is splitting an encyclopedia into A-M and N-Z volumes. Each volume lives on a different shelf. To find 'Apple', you go to volume A-M.

## Key concepts

### `Primary-replica`

One primary accepts writes. Replicas copy the data. Reads can go to any replica. Replicas lag milliseconds behind.

### `Sharding by what?`

By user ID, by region, by date. Pick a key that distributes evenly and matches your query patterns.

### `Hot shards`

One shard gets way more traffic (e.g., celebrity users). Solve with consistent hashing with virtual nodes, or rebalance.

### `Joins across shards`

Expensive. Often avoided by denormalizing data (copying it where it needs to be).

```
# === Replication: same data on multiple servers ===

#    [App]
#      |
#      +--writes--> [Primary DB]
#                       |
#                       +--async replication-->
#                              |
#                              v
#                       [Read Replica 1]
#                       [Read Replica 2]
#                       [Read Replica 3]
#
# App routes writes to primary, reads to any replica.
# Risk: replication lag. Replica might be 100ms behind primary.

# === Sharding: split data across servers ===

# Shard by user_id (mod 4):
#   user_id=1 → shard 1
#   user_id=2 → shard 2
#   user_id=3 → shard 3
#   user_id=4 → shard 0
#   user_id=5 → shard 1
#
# Each shard is a separate database with a subset of users.
# App computes the shard, routes the query.

# === Consistent hashing (better than mod) ===
# Maps both servers and keys to a hash ring.
# When a server is added/removed, only ~1/N of keys move.
# Virtual nodes: each server has multiple positions on the ring
#   → better distribution, less hot-spotting.

# === Resharding ===
# Problem: as you grow, you need more shards.
# Solutions:
#   - Hash-based: just add more shards, keys redistribute
#   - Directory-based: lookup table maps key → shard (extra hop)
#   - Rebalance: read all data, repartition, write back (downtime or dual-write)

# === Cross-shard queries are painful ===
# SELECT * FROM orders WHERE user_id = 1 AND status = 'shipped';
# If user_id and status are on different shards, you need a scatter-gather:
#   - Query all shards
#   - Merge results
#   - Slow. Avoid when possible.

# === When NOT to shard ===
# Most apps never need to shard. If you're not at 1TB+ of data or 10K+ writes/sec,
# you're probably over-engineering. Try read replicas, better queries, more indexes first.

# === Common database scaling journey ===
# 1. Single primary, vertical scaling (bigger server)
# 2. Add read replicas
# 3. Add caching (Redis)
# 4. Archive old data
# 5. Vertical scaling hits a ceiling → shard
```

## Common pitfalls

- ⚠️ Sharding too early. It makes your code more complex. Most apps never need it.
- ⚠️ Choosing a bad shard key. If you shard by `country` and 80% of users are in India, that shard is overloaded.
- ⚠️ Forgetting to update connection strings. Old app servers still write to the old primary after failover.
