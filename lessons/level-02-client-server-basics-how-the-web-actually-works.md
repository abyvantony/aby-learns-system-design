# Level 2: Client-server basics: how the web actually works

**Track:** System Design · **Level:** 2/12 · **Difficulty:** `beginner`

📚 **Today's lesson** — published 2026-06-27

## TL;DR

Every web system has a client (browser, app) and a server. The client sends requests, the server sends back responses. Understanding this is the foundation for everything else.

## Real-world analogies

- The client-server model is a restaurant. You're the client (place an order). The kitchen is the server (prepares food). A waiter (HTTP) carries messages between you.
- HTTP is a language. The client speaks it to ask for things. The server speaks it to answer. They have a structured conversation: request → response → done.

## Key concepts

### `HTTP methods`

GET (fetch), POST (create), PUT (replace), PATCH (partial update), DELETE (remove). Each has a specific purpose.

### `Status codes`

200 OK, 201 Created, 301 Moved Permanently, 400 Bad Request, 401 Unauthorized, 404 Not Found, 500 Server Error. The code tells you what happened.

### `Headers`

Metadata about the request/response. `Content-Type: application/json` says what's in the body. `Authorization: Bearer xyz` proves who you are.

### `Statelessness`

Each HTTP request is independent — the server doesn't remember the previous one. State is carried in cookies, tokens, or the URL.

```
# A simple HTTP exchange

# === Client sends a request ===
# GET /index.html HTTP/1.1
# Host: example.com
# User-Agent: Mozilla/5.0
# Accept: text/html
# (blank line ends headers)

# === Server sends a response ===
# HTTP/1.1 200 OK
# Content-Type: text/html
# Content-Length: 1234
# (blank line)
# <!DOCTYPE html>
# <html>...

# === Real example with curl ===
# $ curl -i https://api.github.com/users/octocat
# HTTP/2 200
# content-type: application/json
# x-ratelimit-limit: 60
#
# { "login": "octocat", "id": 583231, ... }

# === Status code categories ===
# 1xx: Informational    (rare, "hold on")
# 2xx: Success          (200 OK, 201 Created, 204 No Content)
# 3xx: Redirection      (301 Moved, 304 Not Modified)
# 4xx: Client error     (400 Bad Request, 401 Unauthorized, 404 Not Found, 429 Too Many Requests)
# 5xx: Server error     (500 Internal, 502 Bad Gateway, 503 Unavailable, 504 Timeout)

# === REST: a way to design HTTP APIs ===
# Resource-oriented URLs:
#   GET    /users          → list users
#   GET    /users/42       → get user 42
#   POST   /users          → create a new user
#   PUT    /users/42       → replace user 42
#   PATCH  /users/42       → update some fields of user 42
#   DELETE /users/42       → delete user 42

# Idempotency: GET, PUT, DELETE are idempotent (safe to retry).
# POST, PATCH are NOT (each call may have a different effect).
```

## Try it yourself

Using `curl`, make a GET request to `https://api.github.com/users/YOUR_USERNAME`. Note the status code, headers, and body. Now try a 404: `curl -i https://api.github.com/users/this-user-doesnt-exist-12345`. What's the status code?

## Common pitfalls

- ⚠️ Confusing 401 (Unauthorized — you need to log in) with 403 (Forbidden — you're logged in but not allowed).
- ⚠️ Treating POST as idempotent. If you retry a POST payment, you might charge the customer twice. Use idempotency keys.
- ⚠️ Sending sensitive data in URLs (GET parameters). They show up in logs. Use POST with body, or HTTPS + headers.
