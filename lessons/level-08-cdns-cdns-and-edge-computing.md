# Level 8: CDNs, CDNs, and edge computing

**Track:** System Design · **Level:** 8/12 · **Difficulty:** `advanced`

📚 **Today's lesson** — published 2026-06-30

## TL;DR

A CDN (Content Delivery Network) is a global network of servers that cache your content close to users. Instead of every user hitting your origin server in Virginia, they hit the nearest CDN edge in Mumbai.

## Real-world analogies

- A CDN is like a chain of convenience stores. Instead of everyone driving to the central warehouse (origin), they grab products from the nearest store (edge).
- Edge computing is the same idea, but for compute, not just content. Run code close to the user — login checks, image processing, AI inference.

## Key concepts

### `Cache hit ratio`

Percentage of requests served from cache. >95% is great. <70% means your content is too dynamic or your TTLs are wrong.

### `Push vs pull CDN`

Push: you upload content to CDN. Pull: CDN fetches from origin on first request. Pull is easier; push is more control.

### `Cache key`

The URL (usually). `cdn.example.com/image.jpg?v=1` and `?v=2` are different cache keys. Use versioning for instant invalidation.

### `Edge functions`

Lambda@Edge, Cloudflare Workers, Fastly Compute. Run JavaScript/WASM at the CDN edge. Auth, redirects, A/B tests.

# User in Tokyo → request to Cloudflare edge in Tokyo

# === Cache headers ===
# Cache-Control: public, max-age=3600
#   - public: can be cached by CDNs and browsers
#   - max-age=3600: cache for 1 hour
#
# Cache-Control: public, max-age=31536000, immutable
#   - For files with hashes in name (style.[hash].css) — cache forever

# Cache-Control: no-store
#   - Sensitive data (banking, auth tokens) — never cache

# === Cache invalidation strategies ===

#    Most reliable, no CDN coordination needed
# 2. Purge by URL: CDN API to delete specific URLs
#    Slow, costs money, can have lag
# 3. Purge by tag (Cloudflare): tag objects, purge by tag
#    Fastest for bulk invalidation
# 4. Short TTLs: cache for 5 minutes, accept up to 5 min staleness
#    Simplest, works for most content

## Try it yourself

Your app has 10M users, 80% in India. Your origin is in AWS us-east-1 (Virginia). How would you set up a CDN? What cache TTLs would you use for: HTML, CSS/JS with hashes, user avatars, API responses?

## Common pitfalls

- ⚠️ Caching personalized content as if it were public. User A sees User B's data. Use Vary: Cookie or auth tokens in cache key.
- ⚠️ Long TTLs on content that changes. Users see stale prices, broken layouts. Use versioned URLs.
- ⚠️ Forgetting to set Cache-Control on assets. Browser re-fetches every time, slowing your site and burning bandwidth.
