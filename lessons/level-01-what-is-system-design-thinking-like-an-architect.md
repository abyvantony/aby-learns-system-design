# Level 1: What is system design? Thinking like an architect

**Track:** System Design · **Level:** 1/12 · **Difficulty:** `beginner`

📚 **Today's lesson** — published 2026-06-27

## TL;DR

System design is the art of figuring out how all the pieces of a software system fit together — servers, databases, caches, queues, and how data flows between them. It's what separates a developer from an architect.

## Real-world analogies

- If coding is laying bricks, system design is the blueprint of the whole building. You can lay perfect bricks and still build a house that falls down without a good blueprint.
- Building a system is like building a city. You need roads (network), power plants (servers), warehouses (databases), and delivery trucks (queues). System design plans how all of them connect.

## Key concepts

### `Functional requirements`

What the system DOES. 'Users can post tweets', 'Search returns results in under 1 second'.

### `Non-functional requirements`

How WELL it does it. Performance, availability, scalability, security, maintainability.

### `Components`

The building blocks: web servers, databases, caches, message queues, load balancers, CDNs.

### `Trade-offs`

Every design choice has a cost. Faster = more expensive. More features = more complex. System design is choosing the right trade-off, not the 'best' one.

```
# The classic system design interview framework: 4 steps
# 1. Clarify requirements (5 min)
#    - What features? What users? What scale?
# 2. High-level design (10 min)
#    - Draw boxes and arrows on a whiteboard
# 3. Deep dive (15 min)
#    - Pick 1-2 components and design them in detail
# 4. Wrap up (5 min)
#    - Bottlenecks? Failure modes? Monitoring?

# Example: Design a URL shortener (like bit.ly)

# STEP 1: Requirements
# Functional:
#   - Given a long URL, return a short one
#   - Given a short URL, redirect to the original
#   - Users can optionally sign in and see their URLs
# Non-functional:
#   - Short URLs should be as short as possible
#   - Redirects must be fast (< 100ms)
#   - 100:1 read-to-write ratio (way more clicks than creates)
#   - Available 99.9% of the time

# STEP 2: High-level design
#
#   [User Browser]
#        |
#        v
#   [Load Balancer]
#        |
#        v
#   [Web Servers] ---> [Cache (Redis)] ---> [Database (PostgreSQL)]
#        |
#        v
#   [Analytics Service]  (separate, batch)

# STEP 3: Deep dive
# - How to generate short codes? Hash + truncate? Base62 random?
# - How to handle hot URLs? Cache them in Redis with 24h TTL.
# - How to scale reads? Read replicas, CDN for static redirects.
# - How to scale writes? Shard the database by hash of short code.

# STEP 4: Bottlenecks
# - Database is single point of failure → add read replicas
# - Cache goes cold → pre-warm with top 1000 URLs daily
# - Hash collisions → append random suffix or retry
```

## Try it yourself

Pick a system you use daily (Instagram, WhatsApp, Uber). Write down 5 functional and 5 non-functional requirements for it. Don't look anything up — just your intuition.

## Common pitfalls

- ⚠️ Jumping to a specific tech ('let's use MongoDB') before understanding the problem.
- ⚠️ Confusing requirements with solutions. 'We need 1M users' is a requirement; 'use Kafka' is a solution.
- ⚠️ Ignoring non-functional requirements until the end. Performance and reliability are design choices, not afterthoughts.
