# Level 9: Consistency, CAP, and distributed systems

📚 **Today's lesson** — published 2026-06-30

## TL;DR

When data lives on multiple machines, you face a fundamental tension: how do you keep them in sync? Strong consistency (always the same answer) is slow. Eventual consistency (eventually the same) is fast but tricky.

## Real-world analogies

- Strong consistency is a phone call. Both parties hear the same thing at the same time. Slow (synchronous).
- Eventual consistency is a chain email. The message eventually reaches everyone, but at different times. Fast (asynchronous) but messy.

## Key concepts

### `CAP theorem`

In a network partition (always), you choose between Consistency and Availability. Most systems choose AP — stay up, accept temporary inconsistency.

### `Eventual consistency`

If you wait long enough, all replicas will agree. Used by DynamoDB, Cassandra, S3, DNS.

### `Vector clocks`

A way to track causality in distributed data. When two updates happen independently, you can detect the conflict.

### `Conflict resolution`

Last-write-wins (simple, lossy), CRDTs (mathematically mergeable), application-level merge (domain-specific).

```
# # Feed updates happen over the next few seconds.
# # User sees their post immediately, followers see it within seconds.

# === CAP theorem in practice ===
#
# CA (Consistency + Availability): single datacenter, no partition
#   Example: PostgreSQL with synchronous replication
#   Reality: networks fail. You'll be partitioned eventually.
#
# CP (Consistency + Partition tolerance): rejects writes during partition
#   Example: etcd, ZooKeeper
#   Use when: correctness matters more than availability (config, leader election)
#
# AP (Availability + Partition tolerance): serves reads/writes during partition
#   Example: DynamoDB, Cassandra
#   Use when: staying up matters more than perfect consistency (most apps)

# === Conflict resolution example: collaborative document ===
# User A and B edit the same paragraph offline.
# Both come online. Server sees two versions.
#
# Option 1: Last-write-wins
#   Whoever saved last wins. Other changes lost. Simple, lossy.
#
# Option 2: Operational Transform (Google Docs)
#   Server transforms B's edit to be relative to A's edit.
#   Both edits applied, no data loss. Complex.
#
# Option 3: CRDTs (Yjs, Automerge)
#   Data structure designed to merge without conflicts.
#   Mathematical guarantee of consistency. Newer, less mature.

# === Quorum reads/writes ===
# Write: ack from majority (e.g., 2 of 3 replicas)
# Read: read from majority, take latest by timestamp
# Together: strong consistency with 3-5 replicas
# Trade-off: latency (wait for majority)
```

## Try it yourself

You're building a shopping cart for an e-commerce site. A user adds items on their phone and laptop. They go offline. How do you sync when they come back online? What consistency model do you pick?

## Common pitfalls

- ⚠️ Picking CP when you need AP. If your shopping cart refuses to update during a network blip, users will abandon carts.
- ⚠️ Confusing 'eventual consistency' with 'no consistency'. It means 'will be consistent, just not right now'.
- ⚠️ Using vector clocks for everything. Most cases are fine with LWW + a few special cases of merge logic.
