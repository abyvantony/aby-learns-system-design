# Level 11: Microservices vs monolith: when to split, when to stay

**Track:** System Design · **Level:** 11/12 · **Difficulty:** `ultra-advanced`

📚 **Today's lesson** — published 2026-07-01

## TL;DR

A monolith is one big codebase. Microservices are many small ones. Microservices are fashionable but harder than they look. Most teams should start with a monolith and split only when they have a real reason.

## Real-world analogies

- A monolith is a family business. Everyone knows each other. Fast decisions, but if the founder leaves, things fall apart.
- Microservices are a corporation. Each department is independent. Slower decisions, but resilient to any one department's failure.

## Key concepts

### `When monolith wins`

Small team (< 10 devs), unclear domain, early product. You don't know what the right service boundaries are yet.

### `When microservices win`

Different scaling needs (one service needs 10x the resources), independent deploys, multiple languages/teams.

### `Service boundaries`

Bounded contexts from DDD. 'User' means different things in auth vs billing — they're different services.

### `The migration path`

Modular monolith → split the most painful part → repeat. Don't go 0 to 100 microservices.

Every line has a comment. Read it slowly.

```
# === Monolith: one codebase, one deploy ===
# /my-app
#   /users
#   /orders
#   /payments
#   /notifications
# All in one repo, one process, one database.
# Pros: simple, fast to develop, easy to refactor
# Cons: one bug can take down everything, hard to scale parts independently

# === Microservices: many small services ===
# /auth-service     (login, sessions)
# /user-service     (profiles, preferences)
# /order-service    (cart, checkout)
# /payment-service  (charges, refunds)
# /notification-svc (email, push, SMS)
# Each has its own repo, process, database.
# Communicate via HTTP, gRPC, or message queues.
# Pros: independent deploys, independent scaling, fault isolation
# Cons: network calls (latency + failure), data consistency is hard, observability is hard

# === Inter-service communication ===
# Synchronous: HTTP REST, gRPC
#   - Simple, easy to debug
#   - Caller waits. If service is down, caller is down.
#
# Asynchronous: message queue (Kafka, RabbitMQ)
#   - Decoupled, resilient
#   - Harder to debug, eventual consistency

# === Service mesh (Istio, Linkerd) ===
# Handles cross-cutting concerns:
#   - Service discovery
#   - Load balancing
#   - Retries, circuit breakers
#   - mTLS encryption
#   - Distributed tracing
# Without you putting this logic in every service.

# === Data consistency across services ===
# Problem: order-service and payment-service have separate DBs.
# If you create an order, then payment fails, what happens?
#
# Solutions:
# 1. Saga pattern: each service does its step, publishes event.
#    If a step fails, run compensating actions.
#    Example: create order → charge payment → if fail, cancel order
#
# 2. Two-phase commit (2PC): all services vote to commit.
#    Slow, fragile, not recommended.
#
# 3. Eventual consistency + reconciliation:
#    Accept that things can be temporarily inconsistent.
#    Run a background job that finds and fixes inconsistencies.

# === When to split (the test) ===
# Only split a service when:
# 1. You need to scale it differently (10x the resources)
# 2. You need to deploy it independently (different release cadence)
# 3. It has clearly different tech requirements (Python service, but this part needs Go)
# 4. Multiple teams are stepping on each other in the same codebase
#
# If none of these are true, stay in the monolith.
```

## Try it yourself

Take a small app you've built. Could it be a microservice? List the pros and cons. Now imagine scaling to 100x users — which parts would need to split first?

## Common pitfalls

- ⚠️ Going microservices from day one. You don't know the right boundaries yet. Build a monolith, learn, split.
- ⚠️ Sharing a database between services. It's a distributed monolith — the worst of both worlds.
- ⚠️ Synchronous chains: A calls B calls C calls D. Latency adds up. If any service is slow, all are slow.
